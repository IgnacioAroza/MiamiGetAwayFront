### 2. Fix: Listings dropdown aparecía en esquina inferior izquierda al primer acceso

**Root cause:** El `<Suspense>` en `AppContent` envuelve `AdminLayout` completo. La primera vez que navegás a cualquier listing (lazy chunk), el componente suspende y el Suspense de `AppContent` reemplaza TODA la UI — incluyendo `DashboardHeader`. React 18 puede preservar el estado `listingsAnchor` durante el ciclo unmount/remount, dejando un ref a un nodo DOM detached. `getBoundingClientRect()` sobre ese nodo devuelve coordenadas incorrectas → dropdown posicionado mal por floating-ui.

**Fixes aplicados:**

- `src/components/admin/shared/AdminLayout.jsx`:
  - Suspense boundary propio alrededor del `<Outlet>` — el lazy loading de rutas admin ya no desmonta el `DashboardHeader`
  - `height: '100vh'` (en vez de `minHeight`) en el outer Box — el layout es exactamente 100vh, `window.scrollY` siempre 0
  - `minHeight: 0` en el inner main Box — permite que el flex child encoja correctamente sin el magic number `calc(100vh - 130px)`

- `src/components/admin/shared/DashboardHeader.jsx`:
  - `useRef` para el anchor del botón Listings (en vez de `e.currentTarget`) — ref estable que no depende del event object
  - `useEffect` que limpia `listingsAnchor` en cada cambio de `location.pathname` — mata cualquier estado stale de transiciones Suspense
  - `anchorOrigin` y `transformOrigin` explícitos en el `<Menu>` — floating-ui no tiene que inferir el punto de anclaje
  - `disableScrollLock` en el `<Menu>` — evita que MUI agregue `overflow: hidden` al body (causaba layout shift durante el cálculo de posición)
  - Eliminado `overflow: hidden` del outer Box de AdminLayout (fix previo)
