## 1. Fix: apartamentos no mostraban las imágenes cargadas

**Síntoma:** al editar un apartamento en `/admin/services`, el modal solo mostraba "N images selected" (texto) sin miniaturas, y no había forma de ver ni borrar imágenes existentes. Autos/yates/villas sí mostraban miniaturas con botón de borrar.

**Causa:** `ApartmentForm.jsx` (`src/components/admin/apartments/ApartmentForm.jsx`) es un formulario separado del que usan los otros 3 tipos de servicio (que pasan por `ServicesPage.jsx` + `ImageUploader.jsx`). Tenía un `<Input type="file">` nativo cuyo `handleImageChange` **pisaba** el array de imágenes con los archivos elegidos (`setFormData(prev => ({...prev, images: files}))`), nunca renderizaba `selectedApartment.images`.

**Fix:** `ApartmentForm.jsx` ahora reusa el componente compartido `ImageUploader`. Se separó el estado en `existingImages` (URLs ya guardadas) y `newImages` (archivos nuevos), igual que el patrón ya usado en `ServicesPage.jsx`.

---

## 2. Fix: el borrado/orden de imágenes no se persistía en el backend

**Síntoma (encontrado durante el fix anterior):** aunque el modal ya mostraba y "borraba" imágenes en el estado local, guardar sin subir un archivo nuevo no cambiaba nada en la base — el backend solo tocaba el campo `images` cuando llegaban archivos nuevos (`req.files.length > 0`), y además lo pisaba entero.

**Fix (requiere el PR del backend):** el contrato cambió — `existingImages` ahora se manda como **un único campo JSON** (`JSON.stringify(array)`) con el array final de URLs a conservar, en el orden deseado. Antes se mandaba como campos `FormData` repetidos, lo cual era ambiguo con 0 o 1 imagen (con 1 sola imagen, el backend recibía un string plano en vez de un array, y con 0 el campo directamente no llegaba). Aplicado en `ApartmentForm.jsx` y `ServicesPage.jsx` (`handleDialogSave`).

Ver `MiamiGetAwayBack` PR #46 para el detalle del lado backend (`ImageService.syncImages`).

## 3. Fix: pantalla en blanco al crear/editar un yate

**Síntoma:** crear o editar un yate desde `/admin/services` dejaba la pantalla completamente en blanco después de guardar.

**Causa:** en `ServicesPage.jsx`, al guardar con éxito se hace `setDialogOpen(false)` y `setYacht(null)` en el mismo render. El `Dialog` de MUI no desmonta su contenido al instante — lo mantiene montado durante la animación de cierre (~225ms). En ese margen, `ServicesPage` re-renderiza `YachtForm` con `yacht=null`, y `YachtForm.jsx` leía `yacht.name`, `yacht.price`, etc. sin protección → `TypeError: Cannot read properties of null` sin Error Boundary → pantalla en blanco.

`CarForm.jsx` ya tenía el guard `if (!car) return null;` que lo protegía de este mismo patrón. `PropertyForm.jsx` (villas) ya usaba `property?.name` (optional chaining), tampoco afectado.

**Fix:** agregado `if (!yacht) return null;` al inicio de `YachtForm.jsx`, mismo patrón que `CarForm.jsx`.

## 4. Feature: reordenar imágenes por drag and drop

Se agregó `@dnd-kit/core` + `@dnd-kit/sortable` + `@dnd-kit/utilities` (no había ninguna librería de DnD instalada). `ImageUploader.jsx` (compartido por apartamentos, autos, yates y villas) ahora permite arrastrar las miniaturas para reordenarlas.

**Restricción de diseño:** el reorder solo funciona **dentro de cada grupo** (imágenes existentes entre sí, imágenes nuevas entre sí) — no se puede arrastrar una imagen nueva a una posición antes de una existente. Esto es porque el backend arma el array final siempre como `[...existentes-en-el-orden-elegido, ...nuevas]` (las nuevas van al final tras subirse); permitir mezclar visualmente daría un orden que no coincide con lo que termina guardado.

## 5. Investigación: recorte de imágenes en `ImageCarousel` (revertido)

Se probó unificar el aspect ratio de `ImageCarousel.jsx` (listado y detalle de servicios) a 16:9 real, sacando el `height` fijo que hacía que `aspectRatio` nunca se aplicara (CSS ignora `aspect-ratio` cuando `width` y `height` ya son valores definidos). Mejoraba la consistencia visual en apartamentos, pero **cortaba las ruedas** en las fotos de autos (el crop centrado de `object-fit: cover` se vuelve más agresivo con una caja más panorámica). Se probó además compensar con `objectPosition` sesgado hacia abajo (`center 65%` también en el carrusel multi-imagen), pero seguía sin convencer.

**Decisión:** se revirtió todo — `ImageCarousel.jsx`, `ServiceList.jsx`, `ServiceDetails.jsx`, `ExperienceDetailPage.jsx`, `InvestmentDetailPage.jsx` quedaron idénticos al estado previo (`height` fijo por breakpoint, sin `aspectRatio` real aplicado). No shippeado.

**Lección para el futuro:** un aspect ratio panorámico uniforme (16:9) no sirve para todos los tipos de foto de este proyecto — las fotos de auto suelen tener el vehículo ocupando casi todo el cuadro (poco margen), mientras que las de edificios/apartamentos tienen de sobra. Si se retoma, probablemente haga falta un ratio distinto por tipo de servicio, no uno global.

## 6. Admin: apartamentos ordenados por más recientes primero

**Pedido real (aclarado después de una vuelta atrás):** el problema no era el listado público (`/services/apartments`, que ya tiene filtros) sino el panel admin — un apartamento cargado el día anterior quedaba al final, obligando a pasar varias páginas para encontrarlo/editarlo.

**Backend (requiere `MiamiGetAwayBack`):** los 4 modelos (`apartment.ts`, `car.ts`, `yacht.ts`, `villa.ts`) tenían `ORDER BY id ASC` hardcodeado. Se agregó soporte opcional a `?sort=recent` (→ `ORDER BY id DESC`) vía un nuevo helper `parseSortOrder` en `utils/pagination.ts`. Default sin el parámetro: sin cambios (`ASC`), para no alterar ningún consumidor existente que no lo pida explícitamente.

**Frontend admin:** se agregó `sort: 'recent'` al pedido de apartamentos en los dos lugares donde el admin los lista (no se tocó cars/yachts/villas, ni el listado público):
- `src/components/admin/apartments/ApartmentList.jsx` (`/admin/apartments`)
- `src/components/admin/services/ServicesPage.jsx` (pestaña Apartments de `/admin/services`)

No se agregó ningún selector visible — es simplemente el nuevo orden por defecto en esas dos pantallas.

## ⚠️ Advertencia operativa importante

El dev server local (`localhost:5173` + backend en `localhost:3001`) apunta a la **base de datos y cuenta de Cloudinary de producción** (`DATABASE_URL` y credenciales de Cloudinary en `MiamiGetAwayBack/.env` NO son locales). Cualquier prueba de borrado/creación/edición desde el admin panel local afecta datos reales del cliente.

Para probar flujos destructivos (borrar imágenes, crear/borrar entidades), usar siempre datos de prueba propios (crear una entidad descartable, ej. "TEST-BORRAR") y limpiar después. No asumir que "local" significa aislado de producción en este proyecto.
